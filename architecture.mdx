## Layouts and role routing

Three layouts wrap the role-specific page trees. Each layout is guarded by a `ProtectedRoute` that enforces allowed roles before rendering.

| Layout | Role(s) | Route prefix | Description |
|---|---|---|---|
| `DashboardLayout` | `Administrador`, `admin`, `administrador` | `/dashboard/admin` | Full collapsible sidebar, top header with user info, desktop and mobile variants via `MobileNavbarAdmin`. |
| `StudentLayout` | `Alumno`, `student`, `alumno` | `/dashboard/student` | Fixed 256 px sidebar on desktop; bottom navigation bar on mobile via `MobileNavbar`. |
| `TeacherLayout` | `Coordinador`, `teacher`, `profesor` | `/dashboard/teacher` | Minimal sidebar (`TeacherSidebar`), top header; mobile bottom nav via `MobileNavbarTeacher`. |

### ProtectedRoute component

`src/components/ProtectedRoute.jsx` is a React Router layout route that gates access by role:

```jsx src/components/ProtectedRoute.jsx
const ProtectedRoute = ({ allowedRoles }) => {
  const { user, loading } = useAuth();

  if (loading) {
    return (
      <div className="min-h-screen flex items-center justify-center bg-slate-50">
        <div className="animate-spin rounded-full h-12 w-12 border-t-2 border-b-2 border-orange-500" />
      </div>
    );
  }

  const isAuthenticated = !!user;
  if (!isAuthenticated) return <Navigate to="/login" replace />;

  const currentRole = user?.rol || user?.user?.rol || Cookies.get('user_role');

  if (allowedRoles && (!currentRole || !allowedRoles.includes(currentRole))) {
    return <Navigate to="/forbidden" replace />;
  }

  return <Outlet />;
};
```

**Behaviour:**
1. While `AuthContext` is still resolving, a spinner is shown.
2. If no authenticated user is found, the visitor is redirected to `/login`.
3. If the user's role is not in `allowedRoles`, they are redirected to `/forbidden`.
4. Otherwise, `<Outlet />` renders the matching child route.

Role values accepted by each guard (from `App.jsx`):

```jsx
// Student guard
<ProtectedRoute allowedRoles={['Alumno', 'student', 'alumno']} />

// Coordinator guard
<ProtectedRoute allowedRoles={['Coordinador', 'teacher', 'profesor']} />

// Admin guard
<ProtectedRoute allowedRoles={['Administrador', 'admin', 'administrador']} />
```

## API interceptor ‚Äî `apiFetch`

All HTTP calls go through `src/interceptors/api.js`. This module exports `apiFetch`, a `fetch` wrapper that:

- Reads `VITE_API_URL` from Vite's environment at module load time.
- Attaches `credentials: 'include'` to every request so the browser sends the HTTP-only JWT cookie automatically.
- Detects `401 Unauthorized` responses and attempts a transparent token refresh via `POST /auth/refresh` before retrying the original request.
- Queues concurrent requests while a refresh is in-flight, replaying them all once the new token is issued.
- If the refresh also fails, removes the role/name/id cookies, shows a toast notification, and redirects to `/login`.

```js src/interceptors/api.js
export const apiFetch = async (endpoint, options = {}) => {
  const isFormData = options.body instanceof FormData;
  const defaultOptions = {
    ...options,
    credentials: 'include',
    headers: {
      ...(isFormData ? {} : { 'Content-Type': 'application/json' }),
      ...options.headers,
    },
  };

  let response = await fetch(`${API_URL}${endpoint}`, defaultOptions);

  if (response.status === 401 && !endpoint.includes('/auth/')) {
    // ... transparent refresh logic
  }

  return response;
};

// Convenience methods
apiFetch.get    = (endpoint, options = {}) => apiFetch(endpoint, { ...options, method: 'GET' });
apiFetch.post   = (endpoint, body, options = {}) => apiFetch(endpoint, { ...options, method: 'POST', body: JSON.stringify(body) });
apiFetch.put    = (endpoint, body, options = {}) => apiFetch(endpoint, { ...options, method: 'PUT',  body: JSON.stringify(body) });
apiFetch.patch  = (endpoint, body, options = {}) => apiFetch(endpoint, { ...options, method: 'PATCH', body: JSON.stringify(body) });
apiFetch.delete = (endpoint, options = {}) => apiFetch(endpoint, { ...options, method: 'DELETE' });
```

Service files in `src/services/` import `apiFetch` and call it with constants from `src/constants/apiRoutes.js`.

## Authentication flow

```
1. User submits credentials on /login
        ‚îÇ
        ‚ñº
2. POST /auth/login  ‚Üí  API sets HTTP-only JWT cookie
                        + non-HTTP-only cookies:
                          user_role, user_name, user_id
        ‚îÇ
        ‚ñº
3. AuthContext reads cookies on mount,
   populates user state
        ‚îÇ
        ‚ñº
4. React Router renders ProtectedRoute
   ‚Äî checks user.rol / user.user.rol
     or falls back to Cookies.get('user_role')
        ‚îÇ
        ‚ñº
5. Role match ‚Üí renders DashboardLayout / StudentLayout / TeacherLayout
   Role mismatch ‚Üí <Navigate to="/forbidden" />
   Not authenticated ‚Üí <Navigate to="/login" />
        ‚îÇ
        ‚ñº
6. All subsequent API calls go through apiFetch
   ‚Äî 401 response ‚Üí silent POST /auth/refresh ‚Üí retry
   ‚Äî Refresh fails ‚Üí clear cookies ‚Üí redirect /login
```

<Note>
  The JWT itself is stored in an HTTP-only cookie and is never accessible from JavaScript. The `user_role`, `user_name`, and `user_id` cookies are readable by the frontend and are the source of truth for routing decisions.
</Note>
