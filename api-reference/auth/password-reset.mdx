---
title: "Password Reset"
description: "Two-step flow to recover account access, plus an in-session password change endpoint."
---

Academia Gema provides two mechanisms for updating a password:

1. **Forgot password flow** — for users locked out of their account. Triggered from the public `/forgot-password` page.
2. **Change password** — for authenticated users who want to update their current password from within the dashboard.

## Forgot password flow

<Steps>
  <Step title="Request a reset link">
    Submit the user's username to `POST /auth/forgot-password`. If the account exists and has a linked email address, the server sends a password-reset link to that address.

    **Endpoint:** `POST /auth/forgot-password`

    **Request body:**

    <ParamField body="username" type="string" required>
      The user's login username (not their email). The `ForgotPassword.jsx` page labels this field "Nombre de Usuario".
    </ParamField>

    **Response:**

    The server always returns a generic success message regardless of whether the account exists, to prevent username enumeration:

    ```
    "Si el usuario existe y tiene un correo, se ha enviado un enlace."
    ```

    **Code example:**

    ```js
    // ForgotPassword.jsx
    const response = await apiFetch.post(API_ROUTES.AUTH.FORGOT_PASSWORD, { username });
    ```
  </Step>

  <Step title="Set a new password">
    The reset email contains a link to `/reset-password?token=<token>`. The frontend extracts the token from the query string and submits it along with the new password.

    **Endpoint:** `POST /auth/reset-password`

    **Request body:**

    <ParamField body="token" type="string" required>
      The reset token extracted from the `?token=` query parameter in the reset link URL.
    </ParamField>

    <ParamField body="newPassword" type="string" required>
      The new password chosen by the user. The user must confirm it in a second field client-side; the two values are compared before submission.
    </ParamField>

    **Response:**

    On success the server returns `200 OK`. The frontend displays a success toast and redirects to `/login`.

    **Code example:**

    ```js
    // ResetPassword.jsx
    const token = searchParams.get('token');
    const response = await apiFetch.post(API_ROUTES.AUTH.RESET_PASSWORD, {
      token,
      newPassword: password
    });
    if (response.ok) {
      toast.success("¡Contraseña actualizada! Ya puedes iniciar sesión.");
      navigate('/login');
    }
    ```
  </Step>
</Steps>

## Change password (authenticated)

Allows an already-authenticated user to change their password directly from their profile settings.

**Endpoint:** `POST /auth/change-password`

All requests to this endpoint must include `credentials: "include"` so the session cookie is sent. The `apiFetch` interceptor handles this automatically.

<ParamField body="currentPassword" type="string" required>
  The user's current password. The server verifies this before applying the change.
</ParamField>

<ParamField body="newPassword" type="string" required>
  The new password to set.
</ParamField>

<Note>
  This endpoint is authenticated. Requests without a valid session cookie will receive `401 Unauthorized` and trigger the automatic token-refresh flow in `api.js`.
</Note>

## API route constants

All password-related routes are defined in `apiRoutes.js`:

```js
AUTH: {
  FORGOT_PASSWORD:  "/auth/forgot-password",
  RESET_PASSWORD:   "/auth/reset-password",
  CHANGE_PASSWORD:  "/auth/change-password",
  REFRESH:          "/auth/refresh",
}
```

## Error responses

| Endpoint | Status | Meaning |
|---|---|---|
| `POST /auth/forgot-password` | `200 OK` | Always returned (even for unknown usernames) |
| `POST /auth/reset-password` | `400 Bad Request` | Token is invalid or has expired |
| `POST /auth/reset-password` | `200 OK` | Password updated successfully |
| `POST /auth/change-password` | `401 Unauthorized` | Not authenticated or session expired |
| `POST /auth/change-password` | `400 Bad Request` | Current password is incorrect |
